# Milestone Template (Source Template)

> This is the source template used to generate repo-specific MilestoneTemplate.md files. It lives in claude-docs and should not be copied directly into a project without customization.

---

## Template Instructions (Do Not Include In Milestones)

This section explains how to create a repo-specific MilestoneTemplate.md and how the versioning/naming system works. Remove this entire section when generating the actual template for a project.

### Versioning Strategy: MAJOR.MINOR.PATCH.BUILD

All projects use four-digit semantic versioning: `MAJOR.MINOR.PATCH.BUILD`

| Digit | When it changes | Example |
|---|---|---|
| **BUILD** | Each task completion within a milestone | 0.0.1.0 -> 0.0.1.1 -> 0.0.1.2 |
| **PATCH** | Each new milestone (BUILD resets to 0) | 0.0.1.N -> 0.0.2.0 |
| **MINOR** | Major feature, significant enhancement, or phase change | 0.0.N.N -> 0.1.0.0 |
| **MAJOR** | Breaking changes, production release | 0.N.N.N -> 1.0.0.0 |

**Development phases:**

| Range | Phase | Notes |
|---|---|---|
| 0.0.x.x | Project setup and early development | First milestone is 0.0.1.0 |
| 0.1.x.x through 0.8.x.x | Active development, pre-MVP | Bump MINOR for significant features. Stay in this range as long as needed. |
| 0.9.x.x | Beta / MVP testing | Viable product in production with real users for testing |
| 1.0.x.x | Production release | First stable public release |

The MINOR digit can go as high as needed during development. 0.8.50.0 or 0.8.100.0 is fine. Don't rush to 0.9 until the product is genuinely ready for beta testing.

### Milestone Naming: Version-Based

**Milestones are named after the version they produce.** The milestone name IS the starting version number.

**How to determine the next milestone name:**

1. Read `package.json` to get the current version (e.g. `0.5.3.1`)
2. Bump PATCH by 1, reset BUILD to 0: `0.5.4.0`
3. The milestone file is named `Milestone0.5.4.0.md`
4. Each task completion within the milestone bumps BUILD: 0.5.4.1, 0.5.4.2, etc.

**Examples:**

| Current Version | Next Milestone | File Name | Task 1 | Task 2 | Task 3 |
|---|---|---|---|---|---|
| 0.0.0.0 (new project) | 0.0.1.0 | Milestone0.0.1.0.md | 0.0.1.1 | 0.0.1.2 | 0.0.1.3 |
| 0.0.1.3 | 0.0.2.0 | Milestone0.0.2.0.md | 0.0.2.1 | 0.0.2.2 | 0.0.2.3 |
| 0.3.12.5 | 0.3.13.0 | Milestone0.3.13.0.md | 0.3.13.1 | 0.3.13.2 | 0.3.13.3 |
| 0.8.99.4 | 0.9.0.0 | Milestone0.9.0.0.md | 0.9.0.1 | 0.9.0.2 | 0.9.0.3 |

**When to bump MINOR instead of PATCH:**
If the milestone represents a significant feature or phase change, bump MINOR and reset PATCH and BUILD. This is a judgment call. For example, moving from auth implementation to the submission system might warrant 0.0.N.N -> 0.1.0.0.

### Milestone 0.0.1.0: Project Setup

The first milestone in any new project is always 0.0.1.0. It covers:
- Initializing the repo (if not already done)
- Creating package.json with version 0.0.1.0
- Setting up the development environment
- Creating the first feature branch
- Verifying the project builds and runs locally

This milestone is generated by the `/project-new` skill.

### Creating a repo-specific template

Every repo that uses the milestone skill system needs its own `tasks/MilestoneTemplate.md`. Customize it based on the repo's structure:

1. **Determine the repo type:**
   - **Multi-repo (UI + API):** Keep both repo sections in Task 0. Version tracking covers both package.json files. Branches created for each repo.
   - **Single repo (API only, UI only, or standalone):** Remove the second repo references. Task 0 creates one branch. Version tracking covers one package.json.
   - **Multi-repo (3+ repos):** Add additional repo sections following the same pattern.

2. **Replace all bracketed placeholders:**
   - `[repo-name]` with actual folder names
   - `[github-root]` with actual path
   - `[production-url]` with actual deployment URLs
   - `[deployment platform]` with actual platform (Cloudflare, Vercel, etc.)
   - `{developer}` can stay as a placeholder since it changes per developer

3. **Adjust Task 0 steps** to match the number of repos and branch naming convention.

4. **Remove sections that don't apply:**
   - Single repo projects don't need multi-repo file lists
   - Projects without databases don't need migration rollback steps
   - Lightweight projects may not need the full appendices

5. **Save the result** to `[project]/tasks/MilestoneTemplate.md`

### What stays the same across all templates

- Version numbering strategy (MAJOR.MINOR.PATCH.BUILD)
- Milestone name = starting version number
- Task 0 always comes first (version check + branch creation)
- Progress summary table format
- Git workflow policy (Claude creates branches only, developer commits)
- Task structure (description, affected files, implementation steps, testing, success criteria)

## Git Workflow Policy

**CRITICAL: Claude Code must NOT commit or push code to GitHub.**

**Claude Code Responsibilities:**

- Write and modify code files
- Run TypeScript compilation to verify changes
- Run tests if requested
- Prepare code for review
- Create feature branches ONLY (git checkout -b, git push -u origin)

**User Responsibilities:**

- Review code changes in GitHub Desktop (or preferred git tool)
- Create commits with appropriate messages
- Push commits to feature branch
- Create pull requests
- Manage branches and merges
- Deploy to production

**The ONLY Git Exception:**
Branch creation for milestones:

- `git checkout -b {developer}-MMDDYY-[VERSION]-[repo-suffix]`
- `git push -u origin {developer}-MMDDYY-[VERSION]-[repo-suffix]`

All other git operations are handled by the developer.
